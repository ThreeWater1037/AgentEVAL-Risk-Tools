# AgentEVAL 使用指南

本文面向使用 AgentEVAL 生成安全测试任务的项目成员。下游攻击器开发者请直接阅读[接入说明](接入说明.md)。

## 1. 先理解默认行为

AgentEVAL 的正式主路径是：

```text
Agent 描述
  -> Tool1 收集能力与证据
  -> Risk Seed
  -> Tool2 生成结构化 Case
  -> execution_bundle.json
  -> 下游执行器
  -> results.json
  -> AgentEVAL 校验并导入结果
```

`agenteval run` 默认在生成 `execution_bundle.json` 后停止。这是正常完成，不是漏跑了执行阶段。

只有显式增加 `--execute-sandbox` 时，AgentEVAL 才会执行内置确定性沙箱。该沙箱用于验证 case 结构、调度和反馈流程，不会代替下游真实攻击器，也不会产生真实 ASR。

## 2. 本地安装

要求：

- Python 3.10 或更高版本。
- Windows PowerShell、Linux shell 或 macOS shell。
- 只有使用 LLM 增强时才需要 DeepSeek API Key。

PowerShell 安装示例：

```powershell
git clone <repository-url>
Set-Location AgentEVAL

python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -e .

agenteval --help
```

如果不安装 editable package，也可以在仓库根目录设置 `PYTHONPATH=src` 后使用 `python -m agenteval.cli`。团队日常使用建议安装后统一使用 `agenteval` 命令。

需要在本机启动 API 时安装：

```powershell
python -m pip install -e ".[api,yaml]"
agenteval serve --host 127.0.0.1 --port 8000
```

## 3. 第一次运行

仓库内置的 `examples/demo.json` 是单个 mock Agent，不依赖网络或 LLM：

```powershell
agenteval run `
  --input examples/demo.json `
  --out runs/quickstart `
  --count 1 `
  --llm off
```

正常完成后重点查看：

```text
runs/quickstart/
  analysis_session.json   本次分析的输入和元信息
  agent_snapshot.json     Tool1 观察到的能力与证据
  risk_seeds.json         Tool1 候选风险面
  generated_cases.json    Tool2 生成的测试用例
  execution_bundle.json   交给下游执行器的稳定交付包
  pipeline_summary.json   本次准备阶段摘要
```

`--count 1` 表示每个非 `candidate` Risk Seed 生成 1 条 case，不是整个任务最多生成 1 条。

重复使用同一个 `--out` 会开始新的 analysis，并清除该目录上一轮的 `run_result.json`、`feedback_summary.json` 等下游状态；需要保留历史时请使用新的输出目录或先归档。

## 4. 编写 Agent 输入文件

主输入是 `AgentAccessDescriptor`。最重要的字段如下：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `agent_ref` | 是 | 目标 Agent 的稳定名称。 |
| `protocol` | 否 | `mock`、`http`、`python` 或 `runner`，默认 `mock`。 |
| `endpoint` | HTTP 必填 | Agent 接收任务的完整 URL。 |
| `method` | 否 | HTTP 方法，默认 `POST`。 |
| `request_template` | 否 | 请求 JSON；其中 `{{prompt}}` 会替换为探针文本。 |
| `response_key` | 否 | 从响应 JSON 取文本的字段，支持点分路径。 |
| `headers` | 否 | 非敏感固定 HTTP header。敏感 header 不应直接写值。 |
| `auth_ref` | 否 | 保存认证密钥的环境变量名。 |
| `auth_header` / `auth_scheme` | 否 | 认证 header 与 scheme，默认 `Authorization` / `Bearer`。 |
| `inspect` | 否 | healthcheck、OpenAPI 或 schema 的探测地址。 |
| `static_artifacts` | 否 | 已知能力、工具 schema、RAG、memory 等静态信息。 |
| `optional_artifacts` | 否 | Tool1 可读取的文件路径或内联文本。 |
| `sandbox_policy` | 否 | 对该目标的安全边界说明。 |
| `defense_config` | 否 | 交给下游的防御方案或对照组配置；结构由比赛团队约定。 |
| `timeout_s` | 否 | 单次调用超时，默认 30 秒。 |

### 4.1 最小 mock 示例

```json
{
  "agent_ref": "Demo_RAG_Agent",
  "protocol": "mock",
  "static_artifacts": {
    "policy": "standard assistant policy",
    "capabilities": {"rag": true},
    "rag": {
      "top_k": 4,
      "source": "demo_vector_store",
      "external_write": true
    }
  }
}
```

`mock` 适合确认流程和输入结构，不能代表真实 Agent 行为。

### 4.2 HTTP Agent 示例

```json
{
  "agent_ref": "Team_HTTP_Agent",
  "protocol": "http",
  "endpoint": "http://127.0.0.1:9000/chat",
  "method": "POST",
  "request_template": {
    "message": "{{prompt}}",
    "session_id": "agenteval-safe-probe"
  },
  "response_key": "answer",
  "auth_ref": "TEAM_AGENT_TOKEN",
  "inspect": {
    "healthcheck": "http://127.0.0.1:9000/health",
    "openapi_path": "http://127.0.0.1:9000/openapi.json"
  },
  "static_artifacts": {
    "capabilities": {"rag": true, "tool": true},
    "tool_schemas": [
      {
        "name": "web_search",
        "description": "Search approved project documents",
        "inputSchema": {
          "type": "object",
          "properties": {"query": {"type": "string"}},
          "required": ["query"]
        }
      }
    ]
  },
  "sandbox_policy": {
    "mode": "safe_probe_only"
  },
  "defense_config": {
    "profile": "competition_baseline"
  }
}
```

`endpoint` 是实际对话接口；healthcheck 和 OpenAPI 建议填写完整 URL，避免与 `/chat` 路径产生歧义。

不要把 Token、Cookie 或 API Key 直接写进 descriptor，因为 `analysis_session.json` 和 `execution_bundle.json` 会持久化输入。可以只写一个 `auth_ref`，由受信任的连接层或下游执行器从环境变量、Secret Manager 中解析。

### 4.3 Python 与 runner

本地 CLI 仍支持：

- `python`：加载受信任 Python callable。
- `runner`：启动受信任的本地子进程。

这两种协议具有本地代码执行能力，只适合可信仓库和隔离环境。新版 HTTP API 默认拒绝 `python`、`runner`，且客户端请求不能自行打开这一能力。

## 5. 主命令

```powershell
agenteval run --input <agent.json> --out <目录> [选项]
```

常用参数：

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `--input` | 必填 | 单个 descriptor 或兼容的目标清单。 |
| `--select` | 空 | 输入包含多个 Agent 时选择一个 `agent_ref`。主命令一次只准备一个 Agent。 |
| `--out` | 必填 | 本次评测产物目录。 |
| `--count` | `1` | 每个非 candidate seed 生成的 case 数。 |
| `--llm` | `off` | `off`、`auto` 或 `on`。 |
| `--execute-sandbox` | 关闭 | 显式运行代理沙箱并写 `run_result.json`。 |

建议比赛主链路始终先使用默认的“只准备交付包”模式：

```powershell
agenteval run --input agent.json --out runs/team-agent --count 1 --llm off
```

如果输入文件包含多个 Agent，必须显式选择：

```powershell
agenteval run `
  --input examples/current_framework_agents.json `
  --select SimpleRAGChatbot `
  --out runs/simple-rag `
  --count 1 `
  --llm off
```

## 6. LLM 模式

| 模式 | 行为 | 适用场景 |
| --- | --- | --- |
| `off` | 所有可选 LLM 阶段关闭，使用确定性规则和 fallback。 | 首次接入、CI、可复现基线。 |
| `auto` | 有 API Key 时启用，没有时确定性运行。 | 本地探索。 |
| `on` | 要求 API Key 已配置；缺失时立即报错。单个增强调用失败时仍按安全 fallback 继续，并在 provenance 中记录。 | 正式 LLM 增强实验。 |

PowerShell 配置示例：

```powershell
$env:DEEPSEEK_API_KEY="<your-key>"
$env:DEEPSEEK_MODEL="<available-model>"

agenteval run --input agent.json --out runs/llm-run --count 1 --llm on
```

API Key 只应通过环境变量或 Secret 注入，不要写入代码、JSON 输入和运行产物。

## 7. 代理沙箱

需要验证 AgentEVAL 自身的 case 调度和反馈闭环时，可以运行：

```powershell
agenteval run `
  --input examples/demo.json `
  --out runs/sandbox-smoke `
  --count 1 `
  --llm off `
  --execute-sandbox
```

代理沙箱具有以下边界：

- 不调用其他团队实现的真实攻击器。
- 不执行真实环境投毒或破坏动作。
- 根据 case 结构、质量分和稳定哈希产生可复现代理阶段。
- `sandbox_attack_success` 只能用于流程测试，不能标注为真实 `real_attack_success` 或 ASR。

## 8. 把结果交回 AgentEVAL

下游读取 `execution_bundle.json`，执行其中的 `cases` 后生成结果。下面只展示一条；实际 `results` 必须覆盖 bundle 中全部 case：

```json
{
  "results": [
    {
      "case_id": "case_seed_xxx_v01_xxxxxxxx",
      "failure_stage": "attack_success",
      "metrics": {
        "real_attack_success": true,
        "latency_ms": 842
      }
    }
  ]
}
```

导入：

```powershell
agenteval import-results `
  --analysis-dir runs/team-agent `
  --results path\to\results.json
```

下游必须为 bundle 中的每条 case 返回且只返回一条结果。导入阶段会根据 `case_id` 找回对应的 analysis、seed 和 case，拒绝缺失、未知或重复 case，并写入规范化 `run_result.json`。反馈摘要写入 `feedback_summary.json`。

详细字段和八类攻击族的 case 结构见[接入说明](接入说明.md)。

## 9. Docker CLI

构建镜像：

```powershell
docker build -t agenteval:latest .
```

使用镜像内置示例：

```powershell
docker run --rm `
  -v "${PWD}\runs:/app/runs" `
  agenteval:latest `
  run --input examples/demo.json --out runs/docker-run --count 1 --llm off
```

使用宿主机自定义 descriptor：

```powershell
docker run --rm `
  -v "${PWD}\runs:/app/runs" `
  -v "${PWD}\agent.json:/app/config/agent.json:ro" `
  --env TEAM_AGENT_TOKEN `
  agenteval:latest `
  run --input /app/config/agent.json --out runs/docker-agent --count 1 --llm off
```

上例只按变量名转发 `TEAM_AGENT_TOKEN`；descriptor 中应使用 `"auth_ref": "TEAM_AGENT_TOKEN"`，不要把变量值写入挂载文件。

如果待测 HTTP Agent 运行在 Docker Desktop 宿主机，descriptor 中使用：

```json
{
  "endpoint": "http://host.docker.internal:9000/chat"
}
```

Linux Docker Engine 可根据部署环境使用同一 compose 网络，或显式配置 `host-gateway`。不要在容器 descriptor 中使用 `127.0.0.1` 指代宿主机。

## 10. Docker API

启动：

```powershell
docker compose up --build -d
Invoke-RestMethod http://127.0.0.1:8000/healthz
```

查看 OpenAPI：

```text
http://127.0.0.1:8000/docs
```

停止：

```powershell

```

compose 会把本地 `runs` 挂载到容器 `/app/runs`。生产或比赛共享部署还应配置认证、请求大小限制、网络访问控制和独立的 Secret 管理。

## 11. HTTP API

### 11.1 创建评测

```powershell
$body = @{
  target = @{
    agent_ref = "Demo_API_Agent"
    protocol = "mock"
    static_artifacts = @{
      capabilities = @{ rag = $true }
      rag = @{ source = "demo"; external_write = $true }
    }
  }
  count = 1
  llm = "off"
  execute_sandbox = $false
} | ConvertTo-Json -Depth 10

$evaluation = Invoke-RestMethod `
  -Method Post `
  -Uri http://127.0.0.1:8000/api/v1/evaluations `
  -ContentType "application/json" `
  -Body $body

$evaluation
```

响应包含：

- `schema_version`
- `evaluation_id`
- `status`
- `analysis_dir`
- `seed_count`
- `case_count`
- `execution_bundle`

新版 API 主要负责生成交付包，不远程触发真实攻击。`execute_sandbox` 应保持 `false`；需要代理沙箱时优先使用本地 CLI 的显式开关。

### 11.2 查询评测

```powershell
$id = $evaluation.evaluation_id
Invoke-RestMethod "http://127.0.0.1:8000/api/v1/evaluations/$id"
```

### 11.3 回传结果

```powershell
$results = @(
  $evaluation.execution_bundle.cases | ForEach-Object {
    @{
      case_id = $_.case_id
      failure_stage = "not_triggered"
      metrics = @{ real_attack_success = $false }
    }
  }
)

$resultBody = @{ results = $results } | ConvertTo-Json -Depth 10

Invoke-RestMethod `
  -Method Post `
  -Uri "http://127.0.0.1:8000/api/v1/evaluations/$id/results" `
  -ContentType "application/json" `
  -Body $resultBody
```

### 11.4 API 安全边界

- 设置 `AGENTEVAL_API_TOKEN` 后，除 `/healthz` 外的请求必须携带 `X-API-Key` 或 `Authorization: Bearer ...`。共享部署应始终设置该变量。
- 默认禁用 `python`、`runner` 输入。
- 默认禁用服务器本地 artifact 路径，避免远程请求读取宿主文件。
- 默认禁用目标 `auth_ref`；只有可信客户端和受控目标同时满足时，才设置 `AGENTEVAL_ALLOW_TARGET_AUTH=1`。
- 不允许客户端通过请求字段打开本地代码执行。
- 不要把服务未认证地暴露到公网。
- descriptor、bundle 和结果中不要包含明文密钥。
- 下游真实攻击器应部署在单独的受控网络和权限边界内。

## 12. Python API

CLI 和 HTTP API 共用 `AgentEval` 编排层。最小调用方式：

```python
from agenteval import AgentEval


engine = AgentEval(run_root="runs")

prepared = engine.prepare(
    "examples/demo.json",
    out_dir="runs/python-run",
    count=1,
    llm="off",
)

print(prepared.evaluation_id)
print(prepared.output_dir / "execution_bundle.json")
print(prepared.execution_bundle()["cases"])

demo_results = {
    "schema_version": "agenteval.results.v1",
    "results": [
        {
            "case_id": case.case_id,
            "failure_stage": "not_triggered",
            "metrics": {"real_attack_success": False},
        }
        for case in prepared.cases
    ],
}

submitted = engine.submit_results(
    prepared,
    demo_results,
)

print(submitted)
```

`prepare()` 只负责 Tool1、Tool2 和交付包；`submit_results()` 负责校验外部结果并写回运行目录。调用者不需要自行拼装 Tool1Analyzer、Tool2Generator 和反馈逻辑。

上例中的 `demo_results` 仅用于演示 Python 回传形状；正式运行必须替换为下游执行器的真实观测。

## 13. 高级调试命令

原有分阶段命令继续用于定位问题：

```text
analyze-agent
generate-cases
run-cases
refine-cases
apply-feedback
evaluate-tool12
```

普通使用者优先使用 `run` 和 `import-results`，避免各阶段默认值不一致。

## 14. 常见问题

### 为什么没有 `run_result.json`？

默认 `run` 只生成交付包。等待下游返回后执行 `import-results`，或仅为流程验证增加 `--execute-sandbox`。

### 为什么 case 数量大于 `--count`？

`--count` 按每个非 candidate seed 计算。多个 seed 会分别生成 case。

### 为什么没有生成 case？

先检查 `risk_seeds.json`。Tool2 会跳过状态为 `candidate` 的 seed；也应检查 snapshot 是否包含对应能力和 evidence。

### 为什么容器连接不到宿主机 Agent？

容器内 `127.0.0.1` 指向容器本身。Docker Desktop 使用 `host.docker.internal`，或把双方放入同一 Docker 网络。

### 可以把 sandbox 指标写进论文 ASR 吗？

不可以。必须由真实执行器实际观察目标行为，并通过 `real_attack_success` 等字段明确回传。
