# AgentEVAL 接入说明

本文档面向后续接手系统的比赛团队，说明如何把待测 Agent 接入 AgentEVAL，如何读取系统输出，以及如何把底层真实执行器接入现有 Tool1/Tool2 流程。

## 1. 系统功能简介

AgentEVAL 当前主流程是：

```text
Agent 访问描述 -> Tool1 风险发现 -> Risk Seed -> Tool2 用例生成 -> Executor 执行 -> Feedback/Report
```

各模块职责如下：

- Tool1：读取待测 Agent 的访问描述、静态材料和可选动态 probe 输出，形成 `AgentSnapshot`，再基于证据规则生成 `RiskSeed`。
- Tool2：读取 `RiskSeed` 和 `AgentSnapshot`，默认走 SIRAJ-style 生成路径，产出结构化 `GeneratedCase`。
- Executor Registry：根据 case 内的 `executor` 字段选择执行器。当前默认是 deterministic sandbox，只验证结构、调度和反馈闭环；真实执行器需要团队按接口补充。
- Feedback：读取执行结果 `run_result.json`，按执行阶段轻量回写 seed 的置信度和状态。

当前系统强调安全沙箱与可复现实验记录。默认输出不能当作真实攻击成功率；真实 ASR 或真实执行结果必须来自真实执行器或人工标注结果导入。

## 2. 系统输入输出

### 2.1 主输入

系统入口输入是 `AgentAccessDescriptor`，对应代码位置：

```text
src/agenteval/schemas.py -> AgentAccessDescriptor
```

核心字段如下：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `agent_ref` | string | 是 | 待测 Agent 的唯一名称，例如 `AutoGen_MagenticOne`。 |
| `protocol` | string | 否 | 接入协议，支持 `mock`、`http`/`https`、`python`/`python_function`、`runner`/`subprocess`。默认 `mock`。 |
| `endpoint` | string/null | 视协议而定 | HTTP Agent 的服务地址。 |
| `method` | string | 否 | HTTP 请求方法，默认 `POST`。 |
| `request_template` | object | 否 | 请求模板，`{{prompt}}` 会被替换成 Tool1 probe 文本或执行任务文本。默认 `{"message": "{{prompt}}"}`。 |
| `response_key` | string/null | 否 | 从响应 JSON 中取哪个字段作为 Agent 文本输出；不填时系统会尽量自动读取常见字段。 |
| `runner` | object/list/null | 视协议而定 | 本地 runner/subprocess 的启动方式。 |
| `python_callable` | object | 视协议而定 | 本地 Python callable 的模块、函数、reset、inspect 配置。 |
| `inspect` | object | 否 | 目标能力探测配置，例如 `healthcheck`、`schema_path`、`openapi_path`。 |
| `optional_artifacts` | list | 否 | 额外静态材料，可传文件路径或对象，用于 Tool1 补充证据。 |
| `static_artifacts` | object | 否 | 已知能力、工具 schema、RAG、memory、multi-agent 等静态信息。 |
| `expected_domains` | list | 否 | 评估用预期风险域标签，不参与真实执行，只用于代理指标。 |
| `metadata` | object | 否 | 业务侧自定义元数据。 |
| `sandbox_policy` | object | 否 | 沙箱策略说明，默认只做安全 probe。 |
| `timeout_s` | number | 否 | HTTP、runner 等调用超时时间，默认 30 秒。 |

最小 mock 输入示例：

```json
{
  "agent_ref": "Demo_RAG_Agent",
  "protocol": "mock",
  "expected_domains": ["prompt_context_injection", "rag_poisoning"],
  "static_artifacts": {
    "policy": "standard assistant policy",
    "capabilities": {"rag": true},
    "rag": {"top_k": 4, "source": "demo_vector_store", "external_write": true}
  }
}
```

HTTP Agent 输入示例：

```json
{
  "agent_ref": "Http_RAG_Agent",
  "protocol": "http",
  "endpoint": "http://127.0.0.1:8000/chat",
  "method": "POST",
  "request_template": {"message": "{{prompt}}", "session_id": "agenteval_demo"},
  "response_key": "answer",
  "inspect": {
    "healthcheck": "/health",
    "openapi_path": "/openapi.json"
  },
  "static_artifacts": {
    "capabilities": {"rag": true, "tool": true},
    "tool_schemas": [
      {"name": "web_search", "description": "Search sandbox docs", "inputSchema": {"query": "string"}}
    ]
  }
}
```

本地 Python callable 输入示例：

```json
{
  "agent_ref": "Local_Python_Agent",
  "protocol": "python",
  "request_template": {"message": "{{prompt}}"},
  "response_key": "response",
  "python_callable": {
    "module_path": "tests/fixtures/fake_direct_agent.py",
    "callable": "handle",
    "inspect": "inspect_agent",
    "reset": "reset"
  }
}
```

本地 runner 输入示例：

```json
{
  "agent_ref": "Local_CLI_Agent",
  "protocol": "runner",
  "runner": {
    "command": ["python", "path/to/agent_runner.py"]

```

批量目标清单也支持 `targets`、`registry`、`agents`、裸列表或映射对象。清单会先被转换为 `TargetProfile`，再转换为 `AgentAccessDescriptor`。


### 2.2 主输出

一次完整运行通常会在某个 analysis 目录下生成以下文件：

| 文件 | 生成阶段 | 说明 |
| --- | --- | --- |
| `analysis_session.json` | Tool1 | 分析会话元信息，包括 `analysis_id`、原始 `agent_access`、connector 类型和沙箱策略。 |
| `agent_snapshot.json` | Tool1 | Agent 能力快照，包括能力摘要、工具 schema、运行时观测和 evidence 索引。 |
| `risk_seeds.json` | Tool1 | Tool1 生成的风险种子列表，是 Tool2 的主要输入。 |
| `generated_cases.json` | Tool2 | Tool2 生成的结构化测试用例，是执行器的主要输入。 |
| `generation_job.json` | Tool2/API | API 生成任务摘要，包含生成数量、有效数量和 case id。 |
| `run_result.json` | Executor | 执行器返回的运行结果列表。 |
| `feedback_summary.json` | Feedback | 执行反馈回写 seed 后的摘要。 |
| `summary.json` | demo/manifest | 多目标运行汇总。 |
| `paper_tables.md` | evaluate/import | 论文式指标表格。 |

`RiskSeed` 关键字段：

| 字段 | 说明 |
| --- | --- |
| `seed_id` | seed 唯一 ID。 |
| `analysis_id` | 所属分析 ID。 |
| `risk_domain` | 风险域，例如 `rag_poisoning`、`tool_output_injection`。 |
| `entry_point` | 风险入口，例如外部工具结果、消息总线、检索上下文。 |
| `evidence_ids` | 支撑该 seed 的证据 ID 列表。 |
| `preconditions` | 触发该风险所需的前置条件。 |
| `attack_goal` | 沙箱安全目标描述。 |
| `recommended_executor` | 推荐真实执行器名称，Tool2 会复制到 case 的 `executor` 字段。 |
| `confidence` | Tool1 置信度。 |
| `status` | `auto_generate`、`review` 或 `candidate`。 |
| `score_detail` | 规则分、动态分、LLM 补充信息、SIRAJ enrichment 等细节。 |

`GeneratedCase` 关键字段：

| 字段 | 说明 |
| --- | --- |
| `case_id` | case 唯一 ID。 |
| `seed_id` | 来源 seed。 |
| `attack_family` | 风险域，通常等于 seed 的 `risk_domain`。 |
| `delivery_mode` | 投放方式，例如 `direct_input`、`environment_poisoning`。 |
| `setup` | 执行前准备参数，真实执行器主要读取这里配置测试夹具、工具返回、检索材料等。 |
| `trigger` | 触发参数，真实执行器据此调用目标 Agent。 |
| `expected_signal` | 预期观测信号，例如 sandbox marker 是否出现、工具结果是否被采纳。 |
| `cleanup` | 清理参数，真实执行器执行后必须尽量恢复环境。 |
| `executor` | 执行器名称，用于 `ExecutorRegistry` 分发。 |
| `quality_score` | Tool2 对 case 的质量评分。 |
| `provenance` | 生成来源、策略、版本、safe marker 等追踪信息。 |
| `validation_result` | schema、安全和 dry-run 校验结果。 |

`RunResult` 关键字段：

| 字段 | 说明 |
| --- | --- |
| `run_id` | 运行记录 ID。 |
| `analysis_id` | 所属分析 ID。 |
| `seed_id` | 对应 seed。 |
| `case_id` | 对应 case。 |
| `failure_stage` | 执行阶段结果，会影响 feedback。 |
| `metrics` | 执行指标，例如 schema 是否有效、真实是否触发、耗时、错误码等。 |
| `feedback` | 执行器自定义反馈，例如实际选用执行器、错误摘要、原始日志路径。 |

推荐 `failure_stage` 值：

| 值 | 语义 | feedback 影响 |
| --- | --- | --- |
| `attack_success` | 预期信号被真实观察到。 | 提高 seed 置信度。 |
| `action_blocked` | 目标有相关响应，但被防护策略阻断。 | 小幅提高 seed 置信度。 |
| `retrieved_not_adopted` | 检索/读取发生，但没有被后续采纳。 | 小幅提高 seed 置信度。 |
| `adopted_no_action` | 内容被采纳，但没有形成后续动作。 | 轻微提高 seed 置信度。 |
| `require_review` | 结果不确定，需要人工判断。 | 降低 seed 置信度。 |
| `not_triggered` | 未触发目标行为。 | 降低 seed 置信度。 |
| `setup_failed` | 执行器或环境准备失败。 | 明显降低 seed 置信度。 |

## 3. 输入接口和需要输入的内容

### 3.1 CLI 输入接口

单目标 Tool1：

```powershell
$env:PYTHONPATH="src"
python -m agenteval.cli analyze-agent --descriptor examples/current_framework_agents.json --agent AutoGen_MagenticOne --out runs/autogen
```

参数：

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `--descriptor` | 是 | 单个 descriptor、`agents` 列表或裸列表 JSON。 |
| `--agent` | 否 | 从 descriptor 列表中选择一个 `agent_ref`。不填则全部分析。 |
| `--out` | 是 | Tool1 输出目录。多 Agent 时会在该目录下按 agent 名建立子目录。 |
| `--no-dynamic-probe` | 否 | 关闭动态 probe，只使用静态材料。 |
| `--llm-evidence` / `--no-llm-evidence` | 否 | 控制 Tool1 静态语义证据抽取。 |
| `--llm-runtime-events` / `--no-llm-runtime-events` | 否 | 控制 Tool1 运行时事件归纳。 |
| `--llm-review` / `--no-llm-review` | 否 | 控制 Tool1 seed review / SIRAJ enrichment。 |

单目录 Tool2：

```powershell
python -m agenteval.cli generate-cases --analysis-dir runs/autogen --count 3 --profile compact
```

参数：

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `--analysis-dir` | 是 | 包含 `agent_snapshot.json` 和 `risk_seeds.json` 的目录。 |
| `--count` | 否 | 每个 seed 生成 case 数，默认 3。 |
| `--profile` | 否 | `compact` 或 `expanded`，默认 `compact`。 |
| `--siraj-prompts` / `--legacy-prompts` | 否 | 默认走 SIRAJ prompts；`--legacy-prompts` 显式回退旧模板路径。 |
| `--llm-variants` / `--no-llm-variants` | 否 | 控制 Tool2 LLM variant/refinement。 |

执行已生成 case：

```powershell
python -m agenteval.cli run-cases --analysis-dir runs/autogen
```

参数：

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `--analysis-dir` | 是 | 包含 `generated_cases.json` 的分析目录。 |

执行反馈：

```powershell
python -m agenteval.cli apply-feedback --analysis-dir runs/autogen
```

批量 manifest 全链路：

```powershell
python -m agenteval.cli run-manifest --manifest examples/current_framework_agents.json --out runs/demo --count 1 --profile compact
```

示例 demo 全链路：

```powershell
python -m agenteval.cli run-demo --out runs/demo --count 1 --no-llm-evidence --no-llm-runtime-events --no-llm-review --no-llm-variants
```

论文式代理评估：

```powershell
python -m agenteval.cli evaluate-tool12 --descriptors examples/current_framework_agents.json --out runs/eval --count 1 --no-llm-review --no-llm-variants
```

导入人工或真实执行器结果：

```powershell
python -m agenteval.cli import-paper-results --input path/to/results.csv --out runs/imported_results
```

## 4. 输出内容和真实执行器接口

真实执行器接入点在：

```text
src/agenteval/experiment.py
```

必须实现的抽象接口：

```python
from agenteval.experiment import CaseExecutor
from agenteval.schemas import GeneratedCase, RunResult


class CaseExecutor:
    name = "case_executor"

    def run(self, analysis_id: str, cases: list[GeneratedCase]) -> list[RunResult]:
        ...
```

注册入口：

```python
from agenteval.experiment import DEFAULT_EXECUTOR_REGISTRY

DEFAULT_EXECUTOR_REGISTRY.register("rag_poison_runner", RealRagExecutor())
```

系统分发逻辑：

```text
RiskSeed.recommended_executor -> GeneratedCase.executor -> ExecutorRegistry.run()
```

也就是说，真实执行器名称必须和 case 里的 `executor` 字段一致。当前规则常见推荐执行器名包括：

| 执行器名 | 主要风险域 |
| --- | --- |
| `prompt_orchestrator` | prompt/context 注入类。 |
| `rag_poison_runner` | RAG 检索上下文类。 |
| `memory_runner` | 持久记忆类。 |
| `tool_output_runner` | 工具输出注入类。 |
| `mcp_runner` | MCP 描述或工具注册类。 |
| `planning_trace_runner` | 规划链路/中间状态类。 |
| `multi_agent_runner` | 多 Agent 消息总线类。 |
| `search_rag_runner` | 搜索叙事/RAG 类。 |
| `sandbox` / `deterministic_sandbox` | 当前默认沙箱回退执行器。 |

真实执行器最小示例：

```python
from agenteval.experiment import CaseExecutor, DEFAULT_EXECUTOR_REGISTRY
from agenteval.schemas import GeneratedCase, RunResult


class RealPromptExecutor(CaseExecutor):
    name = "prompt_orchestrator"

    def __init__(self, agent_client):
        self.agent_client = agent_client

    def run(self, analysis_id: str, cases: list[GeneratedCase]) -> list[RunResult]:
        results: list[RunResult] = []
        for case in cases:
            try:
                trigger = case.trigger
                user_task = str(trigger.get("user_task", ""))
                attack_prompt = str(trigger.get("attack_prompt", ""))
                response = self.agent_client.send(user_task + "\n" + attack_prompt)
                expected_keyword = str(case.expected_signal.get("keyword", ""))
                observed = bool(expected_keyword and expected_keyword in response)
                stage = "attack_success" if observed else "not_triggered"
                metrics = {
                    "schema_valid": bool(case.validation_result.get("schema_valid", True)),
                    "quality_score": case.quality_score,
                    "real_attack_success": observed,
                }
                feedback = {
                    "mode": "real_executor",
                    "selected_executor": self.name,
                    "expected_signal": case.expected_signal,
                }
            except Exception as exc:
                stage = "setup_failed"
                metrics = {
                    "schema_valid": bool(case.validation_result.get("schema_valid", True)),
                    "quality_score": case.quality_score,
                    "real_attack_success": False,
                    "executor_error": str(exc),
                }
                feedback = {
                    "mode": "real_executor",
                    "selected_executor": self.name,
                    "error": str(exc),
                }

            results.append(
                RunResult(
                    run_id=f"run_{case.case_id}",
                    analysis_id=analysis_id,
                    seed_id=case.seed_id,
                    case_id=case.case_id,
                    failure_stage=stage,
                    metrics=metrics,
                    feedback=feedback,
                )
            )
        return results


DEFAULT_EXECUTOR_REGISTRY.register("prompt_orchestrator", RealPromptExecutor(agent_client))
```

如果 case 的 `executor` 没有注册，系统会自动回退到 deterministic sandbox，并在 `RunResult.feedback` 中写入：

```json
{
  "requested_executor": "rag_poison_runner",
  "selected_executor": "deterministic_sandbox",
  "fallback_reason": "executor_not_registered"
}
```

## 5. 接口如何接入和需要的参数

### 5.1 接入待测 Agent

团队需要先决定待测 Agent 暴露方式：

| 目标形态 | 建议协议 | 必需参数 |
| --- | --- | --- |
| 远端或本地 HTTP 服务 | `http` | `agent_ref`、`endpoint`、`request_template`、可选 `response_key`。 |
| Python 函数式 Agent | `python` | `agent_ref`、`python_callable.module_path` 或 `python_callable.module`、`python_callable.callable`。 |
| CLI/脚本型 Agent | `runner` | `agent_ref`、`runner.command`、`request_template`。 |
| 暂无真实 Agent，只跑流程 | `mock` | `agent_ref`、`static_artifacts`、`expected_domains`。 |

推荐先从 mock 或 python 接入跑通，再换成 HTTP 或真实 runner。

### 5.2 接入真实执行器

真实执行器需要读取的参数来自 `GeneratedCase`：

| 来源字段 | 执行器用途 |
| --- | --- |
| `analysis_id` | 一次实验的全局 ID，用于日志目录和结果归档。 |
| `case.case_id` | 单条 case ID，用于 run id、日志文件名和结果回填。 |
| `case.seed_id` | 关联 Tool1 风险 seed，反馈阶段需要用它聚合结果。 |
| `case.attack_family` | 风险域，执行器可据此选择不同执行策略。 |
| `case.delivery_mode` | 投放方式，决定是否需要先写入环境、工具 fixture、检索材料等。 |
| `case.setup` | 执行前准备参数。 |
| `case.trigger` | 触发目标 Agent 的请求参数。 |
| `case.expected_signal` | 判断是否触发的观测规则。 |
| `case.cleanup` | 执行后的清理动作参数。 |
| `case.validation_result` | Tool2 预校验结果，执行器可拒绝无效 case。 |

真实执行器必须返回 `RunResult` 列表，且列表长度建议和输入 case 数一致。

建议真实执行器额外记录：

- `metrics.real_attack_success`：真实执行是否观察到预期信号。
- `metrics.latency_ms`：目标执行耗时。
- `metrics.schema_valid`：继承 case 的 schema 校验结果。
- `metrics.quality_score`：继承 case 的质量评分。
- `feedback.mode`：写 `real_executor`。
- `feedback.selected_executor`：真实执行器名称。
- `feedback.log_path`：如有详细日志，写相对路径。
- `feedback.error`：异常时写错误摘要。

### 5.3 接入位置建议

为了避免直接修改核心逻辑，建议新增一个执行器模块，例如：

```text
src/agenteval/executors/real_prompt.py
src/agenteval/executors/real_rag.py
src/agenteval/executors/register.py
```

然后在应用启动或 CLI 执行前导入注册模块：

```python
# src/agenteval/executors/register.py
from agenteval.experiment import DEFAULT_EXECUTOR_REGISTRY
from .real_prompt import RealPromptExecutor
from .real_rag import RealRagExecutor


def register_real_executors(config):
    DEFAULT_EXECUTOR_REGISTRY.register("prompt_orchestrator", RealPromptExecutor(config.prompt))
    DEFAULT_EXECUTOR_REGISTRY.register("rag_poison_runner", RealRagExecutor(config.rag))
```

如果后续需要 CLI 原生加载真实执行器，可以再加一个独立入口，例如 `run-cases-real` 或在现有 `run-cases` 中增加 `--executor-config`。当前代码已经预留 registry，不需要改 Tool1/Tool2 数据结构。

### 5.4 真实执行器配置建议

建议单独维护一个执行器配置文件，不要写死在 case 中：

```json
{
  "prompt_orchestrator": {
    "endpoint": "http://127.0.0.1:8000/chat",
    "timeout_s": 30,
    "response_key": "answer"
  },
  "rag_poison_runner": {
    "agent_endpoint": "http://127.0.0.1:8000/chat",
    "fixture_endpoint": "http://127.0.0.1:8000/test-fixtures/rag",
    "cleanup_endpoint": "http://127.0.0.1:8000/test-fixtures/cleanup",
    "timeout_s": 60
  }
}
```

case 负责描述“测什么”，真实执行器配置负责描述“怎么连目标环境”。
